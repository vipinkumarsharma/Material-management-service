How it works

db-risk-guard needs data about your database to analyze your code. Instead of configuring everything manually, you can export 3 CSV files from MySQL and let the tool auto-generate all config files.

Required Query 1: Large Tables

▼

Lists all tables with more than 1 million rows from information_schema.TABLES. Used by --sync to generate large-tables.yml with auto-assigned risk levels.

SQL
SELECT TABLE_NAME, TABLE_ROWS FROM information_schema.TABLES WHERE TABLE_SCHEMA = DATABASE() AND TABLE_ROWS > 1000000 ORDER BY TABLE_ROWS DESC;
📄
Save result as:
tables.csv
Drop into: db-risk-guard/data/tables.csv
ColumnDescriptionUsed for
TABLE_NAMEName of the database tableAdded to large-tables.yml
TABLE_ROWSApproximate row count (from InnoDB statistics)Risk level auto-assignment
Tip

TABLE_ROWS from information_schema is an estimate for InnoDB tables. For exact counts, use SELECT COUNT(*) FROM table_name, but that's slow on large tables. The estimate is good enough for risk classification.

Required Query 2: Indexes

▼

Lists all indexes across all tables from information_schema.STATISTICS. Used by --sync to generate index-metadata.json. The tool uses this to check if your queries have proper index coverage.

SQL
SELECT TABLE_NAME, INDEX_NAME, NON_UNIQUE, SEQ_IN_INDEX, COLUMN_NAME FROM information_schema.STATISTICS WHERE TABLE_SCHEMA = DATABASE() ORDER BY TABLE_NAME, INDEX_NAME, SEQ_IN_INDEX;
📄
Save result as:
indexes.csv
Drop into: db-risk-guard/data/indexes.csv
ColumnDescriptionUsed for
TABLE_NAMETable the index belongs toGroups indexes by table
INDEX_NAMEName of the index (e.g., PRIMARY, idx_user_email)Index identification
NON_UNIQUE0 = UNIQUE index, 1 = non-uniqueisUnique flag in JSON
SEQ_IN_INDEXPosition of column within the index (1, 2, 3...)Column ordering (leftmost-prefix matching)
COLUMN_NAMEColumn name in this position of the indexIndex coverage checking
Why column order matters

MySQL uses leftmost-prefix matching. For a composite index on (a, b, c), a query filtering on WHERE a = ? AND b = ? uses the index, but WHERE b = ? AND c = ? does not. The SEQ_IN_INDEX column preserves this order.

Optional Query 3: Slow Queries

▼

Lists queries slower than 700ms from performance_schema.events_statements_summary_by_digest. Used by --advise to cross-reference slow queries with your index data and generate specific CREATE INDEX recommendations.

SQL
SELECT DIGEST_TEXT, COUNT_STAR, AVG_TIMER_WAIT / 1000000000 AS AVG_MS, SUM_ROWS_EXAMINED, SUM_ROWS_SENT, FIRST_SEEN, LAST_SEEN FROM performance_schema.events_statements_summary_by_digest WHERE SCHEMA_NAME = DATABASE() AND AVG_TIMER_WAIT > 700000000 ORDER BY AVG_TIMER_WAIT DESC LIMIT 100;
📄
Save result as:
slow_queries.csv
Drop into: db-risk-guard/data/slow_queries.csv
ColumnDescriptionUsed for
DIGEST_TEXTNormalized SQL query text (parameters replaced with ?)Parsed by SQL analyzer
COUNT_STARHow many times this query was executedFrequency context in report
AVG_MSAverage execution time in millisecondsSeverity ranking
SUM_ROWS_EXAMINEDTotal rows scanned across all executionsDetects high scan:return ratio

Prerequisites

performance_schema must be enabled on your MySQL server. Check with: SHOW VARIABLES LIKE 'performance_schema';
If it shows OFF, ask your DBA to enable it (requires MySQL restart).

Adjusting the threshold

The 700000000 value = 700ms in picoseconds (MySQL's internal unit). Adjust this based on your SLA:
• 500ms: use 500000000
• 1 second: use 1000000000
• 2 seconds: use 2000000000

Guide How to Export CSV from Your MySQL Client

▼
1Paste the SQL query in the SQL Editor and run it (Ctrl+Enter)
2Right-click on the result grid → Export Data
3Select format: CSV
4Set the output file path to db-risk-guard/data/tables.csv (or indexes.csv, etc.)
5Click Proceed. Done!
DBeaver Tip

Make sure "Include column names" (header row) is checked in the export dialog. db-risk-guard needs the header row to identify columns.

1Run the SQL query in the console (Ctrl+Enter)

Use the mysql command-line client with tab-separated output:

Bash
# Tables export mysql -u root -p -D your_database -e " SELECT TABLE_NAME, TABLE_ROWS FROM information_schema.TABLES WHERE TABLE_SCHEMA = DATABASE() AND TABLE_ROWS > 1000000 ORDER BY TABLE_ROWS DESC;" | tr '\t' ',' > db-risk-guard/data/tables.csv # Indexes export mysql -u root -p -D your_database -e " SELECT TABLE_NAME, INDEX_NAME, NON_UNIQUE, SEQ_IN_INDEX, COLUMN_NAME FROM information_schema.STATISTICS WHERE TABLE_SCHEMA = DATABASE() ORDER BY TABLE_NAME, INDEX_NAME, SEQ_IN_INDEX;" | tr '\t' ',' > db-risk-guard/data/indexes.csv # Slow queries export (optional) mysql -u root -p -D your_database -e " SELECT DIGEST_TEXT, COUNT_STAR, AVG_TIMER_WAIT/1000000000 AS AVG_MS, SUM_ROWS_EXAMINED, SUM_ROWS_SENT, FIRST_SEEN, LAST_SEEN FROM performance_schema.events_statements_summary_by_digest WHERE SCHEMA_NAME = DATABASE() AND AVG_TIMER_WAIT > 700000000 ORDER BY AVG_TIMER_WAIT DESC LIMIT 100;" | tr '\t' ',' > db-risk-guard/data/slow_queries.csv
Note

The tr '\t' ',' converts MySQL's tab-separated output to CSV. You can also skip this step — db-risk-guard auto-detects both tab and comma delimiters.

After exporting all CSVs

  1. 📁
    Verify your files are in place
    Check that db-risk-guard/data/ contains: tables.csv, indexes.csv, and optionally slow_queries.csv
  2. ⚡
    Run sync to generate configs
    java -jar db-risk-guard.jar --sync
  3. 🔍
    Run the source code scan
    java -jar db-risk-guard.jar -s src/main/java -c db-risk-guard/large-tables.yml --index-json db-risk-guard/index-metadata.json
  4. 💡
    Get slow query recommendations (optional)
    java -jar db-risk-guard.jar --advise
